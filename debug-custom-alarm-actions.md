[OPEN] custom-alarm-actions

Symptoms
- For a new custom alarm, the first notification appears without action buttons.
- Tapping that first notification opens the app and stops the alarm.
- After that, a second notification appears with `I Took It` and `Snooze 30 Min`.
- Tapping those action buttons does not appear to trigger terminal/debug output or behavior changes.

Hypotheses
- H1: The first notification is generated by the native `alarm` package, while the action notification is only generated later from the app-side tap/open flow.
- H2: The action notification buttons are shown, but their callback is not registered or not reached in the isolate/state used for this custom-alarm path.
- H3: The action callback is reached, but Android is launching a UI path instead of the background handler, so terminal/debug logging is not emitted where expected.
- H4: The alarm is already stopped by the first-notification tap, so the later action buttons operate on stale notification state and no longer map to an active alarm record.
- H5: Custom slot alarms use a different storage/payload path than normal alarms, causing action-button routing to fail after the replacement notification is shown.

Plan
- Add instrumentation only around the custom/group alarm notification replacement flow and notification action callback path.
- Reproduce once and inspect debug logs to see which notification path fires first and whether action callbacks execute.
- Apply the minimal fix only after the evidence confirms the failing hop.
